Summary

What changed, in one paragraph

Before, freshrss held a static Postgres password that lived forever — a standing liability. Now OpenBao's database secrets engine mints a throwaway role (v-freshrss-…) valid for about an hour and drops it automatically. External Secrets Operator couriers each fresh credential into a Kubernetes Secret; a PgBouncer sidecar lets the app keep one stable local connection while only the pooler's database-side credential rotates — so a leaked credential is worthless within the hour, and rotation costs zero downtime. A continuous probe fired 60 requests straight through a forced rotation and every one returned 200.

Architecture

How a credential reaches the pod — and rotates without a restart

The app never talks to OpenBao. It connects to a pooler over loopback with a credential that never changes; the pooler holds the rotating one.

The hard part

Four bugs — every one caught in isolation, not on the live app

The pipeline was the easy 90%. The pooler pod was a minefield: each of these would have taken freshrss down (and did, twice, before switching to a throwaway test harness that mounted the real secrets with zero connection to the live app).

#SymptomRoot causeFix
1mint failed: permission denied to grant role "freshrss"PG16+ requires ADMIN OPTION to grant a role; CNPG's inRoles only grants plain membershipone-time GRANT freshrss TO vault_admin WITH ADMIN OPTION
2pgbouncer crash-loop: should not run as rootran as uid 0 to write the socket dir; pgbouncer refuses rootrun as uid 70 + pod fsGroup 70 for the emptyDir
3reloader silently never firedFlux envsubst blanked the script's $cur/$BACKEND — a latent zero-downtime failure flux-local can't catchescape shell vars as $$
4pooler→DB: password authentication failed for user "v-…"reading the dynamic path as two data[] entries minted two creds — username from one, password from anothersingle dataFrom.extract (one mint, matched pair)

Lesson banked: never iterate a brand-new sidecar against a single-replica live app — each mistake is an outage. Prove the pod in an isolated harness first, then do one clean cutover. That's exactly how the final cutover went in on the first try.

Security posture

The honest trade — not "100% secure," but understood

Wins
  • App credential is short-lived (~1h), per-workload, auto-revoked.
  • Non-exfiltratable from the app — it holds only the loopback pooler password, never a reusable DB credential.
  • Attributable — every session is a distinct v-… role in the audit log.
Costs
  • A new long-lived crown jewel: vault_admin.
  • shareProcessNamespace slightly weakens app↔pooler isolation.
  • More moving parts = more surface (you saw four bugs).

Roll out to the rest — by workload shape

Key insight: zero-downtime is mostly free. Apps that can run two pods during a restart (multi-replica / stateless) get zero-downtime rotation from an ordinary rolling restart — no pooler. Only single-replica RWO apps (like freshrss) need the sidecar.

Skipsearxng (Valkey)—no auth configured → nothing to rotate

Out of scope — and honestly so: at-rest encryption keys (n8n, authentik, dependency-track, invoiceninja, harbor) are un-rotatable — regenerating corrupts stored data — and OpenBao's transit engine is not a workaround because none of those apps support an external KMS. They stay as pinned local keys.
