hread Digest: Upgrading Talos + kubectl to v1.13.4 (and why node drains stall)
One-line summary: Bumped Talos v1.13.2/.3 → v1.13.4 (kubectl/k8s already latest), discovered the documented node-upgrade flow stalls because talosctl upgrade's built-in drain can't evict single-replica PDB workloads, recovered the storage churn, and explained the recurring spegel drift warning.

Approx date / status: 2026-06-22 → 2026-06-23 — done (Talos upgrade complete on all 5 nodes; spegel drift left as benign/optional-fix).

[REFERENCE] Talos/k8s version pins and where they live
Type: REFERENCE
Verification: [VERIFIED]
What: The source-of-truth version pins are split across two files. talos/talenv.yaml holds talosVersion and kubernetesVersion (with Renovate datasource comments). .mise.toml holds the client tool pins. As of this thread: Talos v1.13.4 (latest patch — kernel 6.18.34, etcd v2.6.12, Go 1.26.4), Kubernetes v1.36.1 (already the newest stable — 1.37 does not exist yet), kubectl 1.36.1, talosctl 1.13.4, talhelper 3.1.11.

[GOTCHA] talosctl upgrade's built-in drain stalls on this cluster's single-replica PDB workloads
Type: GOTCHA
Verification: [VERIFIED]
What: task talos:upgrade-node IP=<ip> (= talhelper gencommand upgrade → talosctl upgrade ... --timeout=10m) does its OWN cordon+drain before reboot. That drain cannot evict single-replica workloads protected by PodDisruptionBudgets (allowedDisruptions: 0). This cluster has many: kyverno-background-controller / kyverno-reports-controller on every node, and ~7 single-instance CloudNativePG DBs on worker-1 (the DB fleet, since worker-1 was added 2026-06-19). The drain loops evicting, hits a 5m global timeout (not the 10m upgrade timeout), errors out, and the upgrade aborts. Net effect: the node is left cordoned, the new image is staged to the inactive partition ("upgrade completed" was printed), but the node never reboots and keeps running the OLD version.
Why it matters: The documented rolling-upgrade flow appears to work but silently stalls; it's not worker-1-specific (kyverno PDBs block every node). Predates worker-1's DB load.
Snippet: failure tail: error draining node "soyo-1": [error when evicting pods/"kyverno-background-controller-..." ... global timeout reached: 5m0s]

[PROCEDURE] Correct per-node Talos upgrade on a PDB-heavy cluster: force-drain first
Type: PROCEDURE
Verification: [VERIFIED] (all 5 nodes reached v1.13.4 this way)
What: Pre-empt the stalling drain by force-deleting pods (DELETE API bypasses PDBs) so talosctl's internal drain finds nothing to block on. Per node, one at a time: force-drain → upgrade → uncordon → verify. CNPG DBs are on the longhorn StorageClass (Immediate binding, no PV node-lock) so they can reschedule; etcd/kubelet are Talos static pods unaffected by kubectl drain, so control-plane quorum is safe.
Why it matters: This is the working alternative to the broken documented flow.
Snippet:

cd ~/projects/webgrip/homelab-cluster
TCF="talos/clusterconfig/talosconfig"
# per node (soyo-1 .20, soyo-2 .21, soyo-3 .22 by IP, fringe .23, worker-1 .24):
mise exec -- kubectl drain <name> --disable-eviction --force --ignore-daemonsets --delete-emptydir-data --timeout=600s
mise exec -- task talos:upgrade-node IP=<ip>
mise exec -- kubectl uncordon <name>
mise exec -- talosctl --talosconfig $TCF -n <ip> version   # confirm Server Tag v1.13.4
mise exec -- kubectl get node <name> -o wide               # Ready, Talos (v1.13.4)
# etcd nodes also: mise exec -- talosctl --talosconfig $TCF --nodes <other-cp-ip> etcd members  # 3/3

[FACT] Distinguish the two Talos node operations and their drain behavior
Type: FACT
Verification: [VERIFIED]
What: Two distinct flows that are easy to conflate: (1) config changes use task talos:apply-node-safe IP=<ip> HOSTNAME=<name> which does an explicit kubectl drain ... --timeout=120s then apply-config; (2) version upgrades use task talos:upgrade-node IP=<ip> which runs talosctl upgrade and relies on Talos's OWN internal drain (you do NOT manually pre-drain in the documented flow, and you must NOT hand-roll the --image string — the recipe looks up talosImageURL + talosVersion itself).
Why it matters: Using the wrong recipe or hand-rolling the image string is an error the user explicitly corrected.
Snippet: .taskfiles/talos/Taskfile.yaml tasks: apply-node-safe, upgrade-node, upgrade-k8s, generate-config

[FACT] Talos upgrade interrupted at drain is safe; check machinestatus STAGE
Type: FACT
Verification: [VERIFIED]
What: When the upgrade drain stalls and you Ctrl+C, the node is fine — it keeps running the old version. Confirm with talosctl ... get machinestatus: STAGE=running, READY=true means stable/no reboot pending (the staged image just hasn't activated). The only side effect to undo is the cordon. A plain talosctl reboot would NOT switch partitions — you must re-run the upgrade (after a clean force-drain) to activate the staged image.
Why it matters: Avoids panic over an etcd/VIP control-plane node that printed "upgrade completed" but still shows the old version.
Snippet: mise exec -- talosctl --talosconfig talos/clusterconfig/talosconfig -n 10.0.0.20 get machinestatus

[FACT] Longhorn post-reboot churn self-heals serially; confinement + key settings
Type: FACT
Verification: [VERIFIED]
What: Rolling-rebooting all nodes left 25 volumes degraded; they self-healed to ~0 over ~1–2h. Longhorn is confined to the two worker nodes (fringe-workstation, worker-1); soyos have allowScheduling=false (ADR-0026), so cordoning a soyo doesn't affect Longhorn replicas. Each volume wants 2 replicas (one per worker); after a worker reboot its replica shows currentState=stopped, healthyAt=NEVER and must rebuild from the other worker. Rebuilds are throttled by concurrent-replica-rebuild-per-node-limit=1 (serial, slow by design) and replica-auto-balance=disabled. Capacity was not a constraint (fringe ~137Gi free, worker-1 ~748Gi free).
Why it matters: Sets expectations: degraded-after-reboot is normal and converges; don't reboot another node until it does (each degraded volume is momentarily single-replica).
Snippet: node-down-pod-deletion-policy=delete-both-statefulset-and-deployment-pod

[GOTCHA] Don't detect Longhorn rebuilds by grepping table output — use JSON rebuildStatus
Type: GOTCHA
Verification: [VERIFIED]
What: kubectl get replicas.longhorn.io ... | grep -ci rebuild always returns 0 — the rebuild state is not a table column; it lives in .status.rebuildStatus (JSON only). Use jq against -o json. Likewise the volume "0 healthy replica" safety check must be computed from replica healthyAt/failedAt.
Why it matters: A false "0 rebuilding" reading nearly led to a wrong "wedged" conclusion when healing was actually progressing.
Snippet:

# robustness histogram
mise exec -- kubectl get volumes.longhorn.io -n longhorn-system -o json | mise exec -- jq -r '[.items[].status.robustness]|group_by(.)[]|"\(length)\t\(.[0])"'
# active rebuilds
mise exec -- kubectl get replicas.longhorn.io -n longhorn-system -o json | mise exec -- jq -r '[.items[]|select(.status.rebuildStatus.state=="in_progress")]|length'
# CRITICAL: any volume with 0 healthy replicas
mise exec -- kubectl get replicas.longhorn.io -n longhorn-system -o json | mise exec -- jq -r '.items|group_by(.spec.volumeName)[]|{v:.[0].spec.volumeName,h:([.[]|select(.spec.healthyAt!="" and .spec.failedAt=="")]|length)}|select(.h==0)|"0-HEALTHY \(.v)"'

[FACT] spegel is the only HR that perpetually shows DriftDetected — root cause
Type: FACT
Verification: [VERIFIED]
What: A global Flux patch sets driftDetection.mode: warn on EVERY HelmRelease (kubernetes/flux/cluster/ks.yaml:42-43) — deliberately detect+log only, never remediate, "without fighting Kyverno mutate webhooks." spegel is the only HR that actually drifts (DaemonSet/kube-system/spegel changed (1 additions, 2 changes, 0 removals), last writer kube-controller-manager). Cause: the spegel chart renders a DaemonSet that omits fields the API server then defaults — confirmed live spec.revisionHistoryLimit: 10 (the "1 addition") and spec.updateStrategy.rollingUpdate.maxSurge: 0 (a "change"), plus a third server-default. Because mode is warn, it's logged every reconcile and never converges. It is benign noise on the FluxResourceDriftDetected alert.
Why it matters: Explains a recurring alert as expected-by-design, not a real fault.
Snippet: fix (per-HR block merges over global patch), in kubernetes/apps/kube-system/spegel/app/helmrelease.yaml:

spec:
  driftDetection:
    mode: warn
    ignore:
      - target: { kind: DaemonSet }
        paths: [/spec/revisionHistoryLimit, /spec/updateStrategy]
