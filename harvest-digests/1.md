[DECISION] Forgejo chosen over GitLab for the homelab forge — rationale
Type: DECISION
Verification: [ASSERTED]
What: For this cluster, Forgejo is preferred over GitLab on four grounds: (1) footprint — Forgejo is a single Go binary (~100–300 MB RAM) vs GitLab's multi-service architecture (Puma, Gitaly, Sidekiq, Workhorse, bundled Postgres/Redis) with a 4 GB floor / ~8 GB realistic minimum, which is untenable on RAM-tight soyo nodes that already OOMKill BestEffort pods; (2) FOSS ethos — Forgejo is nonprofit, fully copyleft (GPL), no open-core, vs GitLab's open-core model where features migrate into paid EE tiers — consistent with prior OSS-driven pivots (off Infisical, off SOPS); (3) GitHub interop — Forgejo Actions is GitHub-Actions-compatible (act runner), enabling workflow reuse on the GitHub→self-hosted migration path, whereas GitLab CI would force a rewrite; (4) best-of-breed already assembled — the cluster already runs Harbor (registry), Renovate (deps), Grafana/Loki/Tempo (observability), so GitLab's main value (bundling those) is redundant.
Why it matters: Documents why the forge choice was made, so it isn't re-litigated; the deciding factor is hardware/RAM budget, not feature parity.

[REFERENCE] When GitLab would actually win
Type: REFERENCE
Verification: [ASSERTED]
What: GitLab is the better choice only when you need the integrated DevOps suite and have hardware to match: complex multi-stage CI (DAG/child pipelines, environments, approval gates), built-in SAST/DAST/dependency/container scanning, compliance/audit features, or a larger team wanting one vendor-supported platform. The one thing Forgejo trades away is CI maturity — GitLab CI is more powerful than Forgejo Actions — but this cluster's CI is GitOps-light (semantic-release cuts releases; Flux does the actual deploy), so heavy pipeline features aren't needed.
Why it matters: Captures the counter-case so the decision is balanced and the trade-off (CI maturity) is explicit.
